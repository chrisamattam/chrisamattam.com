The following is a rough PRD. I used this framework to outline what I wanted to build for the Running sub-section of Hobbies on this website.

## What is the goal?

This product is a feature on the personal website of Chris Mattam. The objective is to ensure that Chris's runs are logged and displayed as a hobby he pursues. The product will enable website visitors to track:

1. The Distance Chris ran last month *(P0)*
2. The Median pace Chris runs at *(P1)*
3. The Number of runs Chris ran last month *(P1)*
4. The Time Chris ran last month *(P2)*
5. Each individual run entry along with the specific metrics of that run as covered above

## What is the problem we are trying to solve?

The personal website has a Hobbies section which has a Running sub-section listed as a hobby, but website visitors have no way to get insights into how active I am with this hobby.

This data and its associated insights are available from two sources — Strava and Fitbit (Google Health).

However, this data contains sensitive personal data which should ideally not be publicly available:

1. Health data such as heart rate, etc.
2. Run map data which shows typical running locations

These concerns lead to the following conclusions:

1. Strava's social tools cannot be used as they require the profile and its associated activities to be public.
2. Fitbit / Google Health has no inherent social sharing features.

## How are we solving this currently?

Currently the runs are tracked either via the Strava app or through my Fitbit health tracker. The run data can be seen through the Strava / Google Health mobile application or through the Strava Web portal.

The run data stored on Fitbit is not exposed to any social circles unless explicitly shared. The data is shared to social circles which are not clearly covered or which cannot be managed — broken user flow.

The run data stored on Strava is available to all profile followers on Strava.

Strava currently offers two types of embeddable widgets. The weekly Summary Widget works well for aggregate stats, but individual activity embeds display map data for each run — a tradeoff we have to work around. The bigger blocker is that this only works if the profile and activities privacy are already set to Public. This takes us back to the tradeoff discussion that prompted the need for this feature.

## What will success look like?

Success will be when:

1. All run data is automatically captured with high accuracy and updated on the website without any human intervention *(P0)*
2. The Running section and individual runs can easily be shared with a distinct link *(P1)*

The north star metric for this product: runs correctly added to the website.

## What are the possible solutions?

1. Embedding Strava widgets directly into the website
2. Building a Strava data pipeline to Google Sheets using Make.com or similar
3. AI Agent which can read data from Google Health

| Solution | Trade Offs | How will it work? |
|---|---|---|
| Strava Widgets | Lack of privacy. Exposing data to the public vs. tracking less data to preserve privacy. | Code to embed the widgets provided by Strava in account settings. |
| Make.com | Multiple points of failure in the automation pipeline. Boring — what are we in 2015? | Create a data pipeline automation through Make.com, send Strava data to a Google Sheet, build a Python workflow to update the website. |
| AI Agent (Google Health) | Token utilization. Data can only be updated monthly. | Create an AI agent and connect it to the Google MCP server. Schedule monthly Takeouts sent to Gmail. Agent reads Gmail through the MCP, finds the email with Google Health takeout data and reads it. The agent opens a PR to update the website with the monthly run data. |

## What needs to be part of the run data?

The following data needs to be extracted from the Google Takeout data as part of the monthly runs summary, along with the month name:

| Data from Google Health | Mapped to on the Website | Comments |
|---|---|---|
| `SUM(distance)` across Run entries where `distance != null` and `pace != null`, `startTime` in target month | Distance Chris ran last month | `distanceUnit` = Kilometer. Runs missing distance/pace are excluded entirely, not just from this metric. |
| `MEDIAN(pace)` across the same filtered set, converted sec/km → min:sec/km | Median Pace of Run | `pace` is Fitbit's precomputed `activeDuration ÷ distance` — already excludes paused time. |
| `COUNT()` of the same filtered set | Number of Runs last month | Only counts runs with both distance and pace recorded — GPS-less auto-detected runs are dropped. |
| `SUM(activeDuration)` across the same filtered set, ms → hh:mm | Total time Chris ran last month | Uses `activeDuration`, not `duration` — `duration` includes paused/stopped time and would inflate the total. |

The following data needs to be extracted from each individual run of the month:

| Data from Google Health | Mapped to on the Website | Comments |
|---|---|---|
| `distance` (+ `distanceUnit`) | Distance Covered | Entry excluded from the site entirely if this is null — no blank/placeholder shown. |
| `pace`, converted sec/km → min:sec/km | Pace of the Run | Verified `pace = 3600 / speed`; both already exclude paused time. |
| `activeDuration`, ms → mm:ss | Time of the Run | Not `duration` (elapsed, includes pauses) — `activeDuration` is true moving time and is the value Fitbit used to compute `pace`, so time and pace stay consistent. |
